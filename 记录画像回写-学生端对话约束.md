# 记录/画像回写--学生端对话约束

本文档用于约束本次对话只处理“学生端记录/画像回写”功能域。后续问题、修改、验收都以这里定义的产品边界和数据链路为准。

## 一、产品定位

“记录/画像回写”不是一个单独的展示页，而是学生端学习闭环背后的数据回写功能域。

它的产品目标是：把学生完成首次诊断、词汇检测、学案学习后产生的结果，写回统一的数据主链，再让首页、任务池、成长画像、我的词汇、学习进度、检测记录、最近学案读取同一套真实数据。

本功能域聚焦：

- 首次诊断结果保存。
- 成长画像快照生成与读取。
- 词汇检测结果保存与学习进度同步。
- 学案、词表、已学单词之间的一致性归档。
- 缓存清理与页面刷新后的状态一致性。
- 暴露写入、同步、读取失败的真实原因，不用 fallback 或静默兜底隐藏问题。

## 二、产品说明依据

本仓库已有协作说明明确：

- `AGENTS.md` 要求跨页面、跨数据层问题必须先统一数据源和状态链，再谈 UI 修饰。
- `WORKSTREAMS.md` 将本功能域定义为“训练结果回写、画像快照、lesson/vocab 数据一致性”。
- `student_shell_view.py` 将“成长画像”定义为展示诊断结果和阶段画像，帮助学生理解学习重点。
- `student_content_service.py` 将已学单词、检测记录、学习进度、最近学案统一从 `student_records_data.py` 和 `db_student.py` 读取。
- 后台“生成链路与状态机”约束强调：保存、学生消费、教师查看、历史记录必须使用同一份真实数据。
- 后台“数据与迁移”约束强调：学生、学案、学案词表、学习进度、检测记录、首次诊断题库等主链数据必须可上线、可追溯、可验收。

因此，本功能域的核心不是“页面上显示几块卡片”，而是保证记录能写进去、画像能由记录生成、所有页面能从同一数据源读出来。

## 三、当前功能事实

当前学生端相关页面和数据层分工如下：

- `student_initial_diagnosis_view.py`
  - 展示首次诊断流程。
  - 调用 `student_diagnosis_service.evaluate_initial_diagnosis()` 计算诊断结果。
  - 调用 `student_records_data.save_initial_diagnosis_result()` 保存诊断记录、词汇诊断明细和画像快照。

- `student_diagnosis_service.py`
  - 定义首次诊断模块。
  - 根据答题结果生成 `scores`、`module_reports`、`priority_module`、`vocab_diagnostic_result`、`next_actions` 等画像原料。

- `diagnostic_vocab_service.py`
  - 负责首次诊断词汇题库导入、校验、抽题和题目 payload 构建。
  - 不负责学生画像写入。

- `student_records_data.py`
  - 学生端记录读取与画像回写主文件。
  - 读取最近学案、学案词表、已学词汇、检测记录、最新诊断记录、最新画像快照。
  - 写入首次诊断记录、词汇诊断结果、词汇诊断答题明细、画像快照。
  - 清理相关缓存，保证页面重新读取最新状态。

- `db_student.py`
  - 负责词汇检测提交。
  - 写入 `vocab_test_records` 和 `vocab_test_record_items`。
  - 同步 `student_vocab_progress`，再触发记录和首页缓存清理。

- `student_home_viewmodel.py`
  - 聚合首页、任务池和成长画像所需数据。
  - 读取诊断记录、画像快照、检测记录、已学单词、学习进度，决定今日任务和画像展示。

- `student_content_service.py`
  - 为最近学案、我的词汇、学习进度、检测记录等页面构建视图数据。

## 四、主数据链路

### 1. 首次诊断到成长画像

1. 学生在 `student_initial_diagnosis_view.py` 完成首次诊断。
2. `student_diagnosis_service.py` 计算诊断结果。
3. `student_records_data.save_initial_diagnosis_result()` 写入：
   - `student_diagnostic_records`
   - `diagnostic_vocab_results`
   - `diagnostic_vocab_answers`
   - `student_profile_snapshots`
4. 写入成功后清理：
   - `student_records_data` 内相关缓存。
   - `student_home_viewmodel` 缓存。
5. 首页、任务池、成长画像重新读取最新画像快照。

### 2. 词汇检测到学习进度和检测记录

1. 学生在词汇检测页提交答案。
2. `db_student.submit_student_test()` 生成逐题结果。
3. 写入：
   - `vocab_test_records`
   - `vocab_test_record_items`
4. 同步：
   - `student_vocab_progress`
5. 更新 `vocab_test_records.is_synced_to_progress`。
6. 清理学生进度、记录、首页 viewmodel 缓存。
7. 我的词汇、学习进度、检测记录读取同一套记录和进度。

### 3. 学案与已学词归档

1. 后台生成链路保存学案和学案词表。
2. 学生端通过 `student_records_data.get_student_recent_lesson_snapshots()` 读取 `lessons`。
3. 学生端解析学案词表并匹配 `vocab_items`。
4. `get_student_learned_vocab_summary()` 将 `student_vocab_progress` 与最近学案词表按 normalized lemma 关联。
5. 我的词汇按学案归档展示，未匹配到具体学案的词进入未关联分组。

## 五、数据表职责

- `student_diagnostic_records`
  - 首次诊断主记录。
  - 保存模块得分、总分、基础画像字段。

- `student_profile_snapshots`
  - 成长画像快照。
  - 用 `source_record_id` 关联诊断主记录。
  - `profile_payload` 保存页面和任务池需要的完整画像结构。

- `diagnostic_vocab_results`
  - 首次诊断词汇维度汇总。
  - 保存 L1-L5、题型、信心、不确定率等指标。

- `diagnostic_vocab_answers`
  - 首次诊断词汇逐题答题记录。
  - 用于后续追溯具体题目、层级、标签和错误结构。

- `vocab_test_records`
  - 一整轮词汇检测主记录。
  - 记录来源、题型、总题数、正确数、正确率和是否同步到进度。

- `vocab_test_record_items`
  - 词汇检测逐题明细。

- `student_vocab_progress`
  - 学生词汇学习进度主表。
  - 我的词汇、学习进度、复习调度都应读取它。

- `lessons`
  - 学案主记录。

- `lesson_vocab_items`
  - 学案词表绑定。
  - 后台保存链路负责写入，学生端只能消费，不应在展示层伪造。

## 六、职责边界

本对话默认可处理文件：

- `student_records_data.py`
- `student_diagnosis_service.py`
- `diagnostic_vocab_service.py`
- 与本功能域直接相关的检查脚本或 Markdown 文档

谨慎处理高冲突文件：

- `db_student.py`
- `student_home_viewmodel.py`
- `student_initial_diagnosis_view.py`
- `student_content_service.py`
- `student_shell_view.py`
- `student_app.py`

如果问题根因在词汇训练交互，只能指出断点并与“学生端-词汇训练与检测”域协调；如果根因在首页/任务池/画像展示，只能指出断点并与“学生端-P0-首页/任务池/画像”域协调；如果根因在后台生成或教师端数据源，不允许在学生端继续堆补丁。

## 七、禁止事项

- 不允许把数据库写入失败显示成“暂无记录”。
- 不允许查询失败时静默返回空数组或空画像。
- 不允许只更新页面文案，不修正记录写入或状态回写。
- 不允许新增一套与 Supabase 主链并行的展示层状态。
- 不允许用 fallback 跳过 `student_diagnostic_records`、`student_profile_snapshots`、`student_vocab_progress` 等主链表。
- 不允许画像快照脱离来源记录生成。
- 不允许词汇检测有分数但不明确暴露历史记录保存或进度同步失败。
- 不允许把学案词表从文本临时解析出的结果当成唯一事实来源，必须与 `vocab_items` / `student_vocab_progress` / `lesson_vocab_items` 的主链关系对齐。
- 不允许保留已经废弃、无人引用、与当前功能路径冲突的旧逻辑。

## 八、排查优先级

遇到记录/画像回写问题，按以下顺序查：

1. 当前页面入口属于首次诊断、词汇检测、成长画像、我的词汇、学习进度、检测记录还是最近学案。
2. 该入口的唯一数据源是哪张表或哪组表。
3. 写入链路是否真的执行，是否拿到了主记录 ID。
4. 主记录、明细记录、画像快照之间是否有关联键。
5. 写入后是否清理了 `student_records_data`、`db_student`、`student_home_viewmodel` 相关缓存。
6. 页面展示是否读取同一套数据，而不是另起临时状态。
7. 如果记录存在但页面不显示，优先查读取条件、排序、缓存和关联键。
8. 如果页面显示但数据库没有，优先查是否存在展示层假状态。
9. 如果学生端无法修复，明确上游断点：后台生成、迁移、教师端、Supabase Schema 或权限。

## 九、验收方式

每次改动完成后必须说明：

- 改动目标。
- 涉及文件。
- 逻辑变化。
- 验收方式。

最小验收至少包含：

- Python 语法检查。
- 学生端符号契约检查：`python scripts/check_student_app_symbols.py`。
- 最小联调检查：`powershell -ExecutionPolicy Bypass -File scripts/dev_smoke.ps1`。
- 如涉及首次诊断：说明验证了 `student_diagnostic_records`、`student_profile_snapshots`、`diagnostic_vocab_results`、`diagnostic_vocab_answers` 的写入或读取链路。
- 如涉及词汇检测：说明验证了 `vocab_test_records`、`vocab_test_record_items`、`student_vocab_progress` 的写入和同步链路。
- 如涉及学案/已学词：说明验证了 `lessons`、`lesson_vocab_items`、`vocab_items`、`student_vocab_progress` 的关联。
- 如无法连接真实 Supabase，必须明确说明未验证的真实数据链风险，不能用本地或 mock 结果替代。

## 十、本对话当前目标

当前目标是先确立记录/画像回写功能域的产品边界和约束，再继续处理具体缺陷或改造。

后续任何代码修改，都必须先判断是否服务以下之一：

- 训练结果回写。
- 画像快照生成与读取。
- lesson/vocab 数据一致性。
- 记录、进度、画像之间的状态链一致性。

不属于以上目标的内容，应转交对应功能域，不在本对话中夹带处理。
